Provisioning Message Diagram

Provisioning is only needed to be performed once. This protocol allows the Nymi and an NEA to establish a relationship that allows them to recognize each other securely in the future. The process is divided into several steps;

  1. Discovery: Initiated through nclStartDiscovery, this starts the discovery process on the NEA and returns an NCL_EVENT_DISCOVERY when a Nymi in provisioning mode is discovered. This is passive process that does not establish a connection with the Nymi

    if (e.type == NCL_EVENT_DISCOVERY){
    //connect to discovered device or collect multiple discovery events
    //and apply a heuristic to connect to desired device (e.g. proximity by RSSI)
    }
    }

    ...
    NclBool res = nclStartDiscovery();

    }
  2. Agree: If a Nymi is discovered, nclAgree can be used to establishes a connection between. The NEA and the Nymi both parties exchange a short-term key, which is displayed through the LEDs of the connected Nymi. It is expected that the NEA user confirms the LED pattern to guarantee that the connected Nymi is the desired Nymi

  3. Provision: If the client is satisfied that the connected Nymi is the desired one, it can proceed to exchange a long-term key to allow them to recognize each other securely in the future. When successfully provisioned, the Nymi will immediately disconnect from the NEA and return to an active state

Storing the provision in persistent memory is necessary. A sample of how to store and retreive provisions is provided in Storing Provisions. Storing the data is a sensitive matter. By sharing the Nymi provision key with other application, you compromise the integrity of the Nymi's trust. Subsequently, the Nymi can independently revoke their ability to communicate with your application.

To quickly get started with this whole process, sample code for each platform on how to provision, and validate a Nymi is provided as sample code. An indepth tutorial of how to get started with the Hello Nymi Tutorial
